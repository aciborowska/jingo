In the present implementation, ds-pwp-character-set is not case-sensitive.

$ ldapmodify \
>  --hostname localhost \
>  --port 1389 \
>  --bindDN uid=admin \
>  --bindPassword password << EOF
> dn: cn=Policy with character set validation,dc=example,dc=com
> objectClass: top
> objectClass: subentry
> objectClass: pwdPolicy
> objectClass: ds-pwp-password-policy
> objectClass: ds-pwp-validator
> objectClass: ds-pwp-character-set-validator
> cn: Policy with character set validation
> pwdAttribute: userPassword
> ds-pwp-default-password-storage-scheme: PBKDF2-HMAC-SHA256
> ds-pwp-allow-unclassified-characters: true
> ds-pwp-character-set: 0:abcdefghijklmnopqrstuvwxyz
> ds-pwp-character-set: 0:ABCDEFGHIJKLMNOPQRSTUVWXYZ
> ds-pwp-character-set: 0:0123456789
> ds-pwp-character-set: 0:!$%^.#
> ds-pwp-min-character-sets: 3
> subtreeSpecification: { base "ou=people", specificationFilter "(uid=bjensen)" }
> EOF
# ADD operation successful for DN cn=Policy with character set validation,dc=example,dc=com

Notice that the ds-pwp-character-set: 0:abcdefghijklmnopqrstuvwxyz configuration is lost on update.

If you set the attribute to use caseExactMatch in the schema, the problem disappears.
