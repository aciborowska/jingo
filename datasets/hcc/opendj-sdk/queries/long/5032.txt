Duplicate entryUUID's are created when adding an entry using another's previously generated entryUUID in its RDN.

Note: the following entryUUID already exists in the backned.

	dn: uid=user.0,ou=People,dc=example,dc=com
	entryUUID: 0d3ce3bf-4107-3b34-9e5a-fa71deb8b504



Create a new entry to add using the above entries entryUUID value.

opendj; bin /$ cat entryUUID
dn: entryUUID=0d3ce3bf-4107-3b34-9e5a-fa71deb8b504,ou=People,dc=example,dc=com
objectClass: top
objectClass: inetOrgPerson
objectClass: organizationalPerson
objectClass: person
givenName: Fake
sn: User
cn: Fake User
userPassword: Some Fake User
description: This is the description for the faked entryUUID user.
uid: 0d3ce3bf-4107-3b34-9e5a-fa71deb8b504

Add the user: 

opendj; bin /$ ./ldapmodify --port 1389 -D "cn=Directory Manager" -j pass EntryUUID
# ADD operation successful for DN entryUUID=0d3ce3bf-4107-3b34-9e5a-fa71deb8b504,ou=People,dc=example,dc=com

A duplicate entryUUID now exists in the system.

opendj; bin /$ ./ldapsearch --port 1389 -D "cn=Directory Manager" -w password --baseDN ou=People,dc=example,dc=com entryUUID=0d3ce3bf-4107-3b34-9e5a-fa71deb8b504 entryUUID
dn: uid=user.0,ou=People,dc=example,dc=com
entryUUID: 0d3ce3bf-4107-3b34-9e5a-fa71deb8b504

dn: entryUUID=0d3ce3bf-4107-3b34-9e5a-fa71deb8b504,ou=People,dc=example,dc=com
entryUUID: 0d3ce3bf-4107-3b34-9e5a-fa71deb8b504

Problem reproduced with versions:

	3.5.3
	4.0.0
	5.5.0
	6.0.0 RC3
