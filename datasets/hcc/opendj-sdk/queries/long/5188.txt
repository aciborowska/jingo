We have lot of DSML tests failed due to some changes in the SOAP response.

Previously we got this:

      <modifyResponse>
         <resultCode code="32" descr="noSuchObject"/>
         <errorMessage>Entry uid=abergin,ou=People,dc=siroe,dc=com cannot be modified because no such entry exists in the server</errorMessage>
       </modifyResponse>


And now this:

      <modifyResponse matchedDN="">
         <resultCode code="32" descr="noSuchObject"/>
         <errorMessage>Entry uid=abergin,ou=People,dc=siroe,dc=com cannot be modified because no such entry exists in the server</errorMessage>
       </modifyResponse>


==> we now provide 'matchedDN' attribute in the response but as this attribute is optional we should only return it when non empty

Same kind of problem with searchResponse:
Previously:

       <searchResponse>
         <searchResultDone>
           <resultCode code="0" descr="success"/>
         </searchResultDone>
       </searchResponse>


And now:

       <searchResponse>
         <searchResultDone>
           <resultCode code="0" descr="success"/>
           <errorMessage/>
         </searchResultDone>
       </searchResponse>


=> not sure this is really helpful to return an empty 'errorMessage' when the operation succeeds
