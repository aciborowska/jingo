Issue raised in Micronaut Gitter and was advised to raise issue by @jameskleeh .

When you have an interface that defines your HTTP API, and another `@Client` interface which extends that API interface, using the `@Client` to execute a method that defines a URL Path Parameter **coupled with** a second `@QueryValue` parameter will not work e.g.:

```
@Get("/path/{someParam}") 
public String getSomething(String someParam, @QueryValue String someOtherParam) //...
```

A work-around is to redefine the same method signature in your `@Client` interface, which is already defined in the parent in the same way (but it seems redundant to have to do that).
To test this work-around, please uncomment the code in TestClient.java from my git repository.

### Steps to Reproduce

1. Clone the provided GIT Repo
2. Run the JUnit Test provided called 'TestControllerTest.java'
3. See that the test fails
4. Uncomment the code in TestClient.java which is the same method definition from the parent interface
5. Re-run the tests -> tests will pass

### Expected Behaviour

I would expect that when my `@Client` extends an existing interface that uses URL Path Parameters coupled with a second `@QueryValue` parameter, that it should work like other `@Get` examples that do not use this combination. I would expect that you shouldn't have to re-define the same method signature in the extending interface for this to work.

### Actual Behaviour

The `@Client` executing the method with the URL Path Parameter and second `@QueryValue` parameter fails
