This issue raises when trying to use tcp join to form a cluster in a cloud foundry environment that supports container networking and has app service discovery support via dynamic DNS. DNS entries for new app instances are created once an app instance has successfully started. See here for a description of its use: https://github.com/cloudfoundry/cf-app-sd-release#app-developer-experience

The instances starts as single member cluster, and establish connection to each other. But they cannot form a cluster. See the logs below.

It looks like the second instance establishes connection to the first one using its IP address (before registering to DNS), and the first instance binds the connection. But after the start up, the first instance couldn't find the established connection it its `connectionsMap` in `TcpIpConnectionManager` since it looks for the hostname.  

The origin of this issue is in this community topic: https://groups.google.com/d/topic/hazelcast/7GlXxcjwSaE/discussion
