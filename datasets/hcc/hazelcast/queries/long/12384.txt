### Description
While a map is being populated from the map store, `IMap.get()` and `IMap.asyncGet()` may not wait for the ongoing load to complete, if the loader `owner` node dies during the load. It is because the replicas have no (currently checked - see the details section below) information about the ongoing load and after the migration, the `new owner` `previously replica` node may execute the operations if they arrive before load requests from the `SENDER`. In this case, the map's backing structure makes a last attempt to load the single key from the map store, which mitigates the issue. Though with improper `MapLoader` implementations - e.g. only bulk loading is implemented, single key loading is not - the operations may return `null` even for existing keys.

This issue may impact the performance of key loading badly if the `new owner` node is being hit extensively by `GetOperation`s before loading requests arrive.

### Details
`GetOperation` waits for `DefaultRecordStore#loadingFutures` to complete before it executes. But these futures are maintained only on the owner node, and because of this `DefaultRecordStore#checkIfLoaded()` doesn't fail the operation, hence the observed issue in edge cases. 

`DefaultRecordStore` also has a `MapKeyLoader`, which tracks the progress of loading, but that's not checked at all in `DefaultRecordStore#checkIfLoaded()`, therefore not failing `GetOperation`.
