This behavior repeats after timeout defined in hazelcast.merge.next.run.delay.seconds property.
This behavior don't reproduced in 3.1.7 and 3.2.6 versions.
It seems it's a bug in the com.hazelcast.cluster.NodeMulticastListener class that came from commit
https://github.com/hazelcast/hazelcast/commit/8de485c44ebe025675ee6a629347301373ca57bc
when in com.hazelcast.cluster.NodeMulticastListener#isValidJoinMessage function was removed following lines:

```
try {
 return node.getClusterService().validateJoinMessage(joinMessage);
 } catch (Exception e) {
 return false;
 }
```
