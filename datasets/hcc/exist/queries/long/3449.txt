**Describe the bug**

When two library modules import each other, unexpected errors are raised by global variable declarations when eXist evaluates the 2nd module. 

**Expected behavior**

When eXIst compiles a library module that has mutual imports with another library module, eXist should not raise erroneous errors about undefined variables.

**To Reproduce**

Create `/db/test` and store the following two modules:

**/db/test/test1.xqm**

```xquery
xquery version "3.1";

module namespace test1 = "test1";

import module namespace test2 = "test2" at "test2.xqm";

declare variable $test1:a := "a";
```

**/db/test/test2.xqm**

```xquery
xquery version "3.1";

module namespace test2 = "test2";

import module namespace test1 = "test1" at "test1.xqm";

declare variable $test2:x := "x";

declare variable $test2:y := $test2:x || "y";
```

After storing these two queries, open test2.xqm and click Submit in Java admin client or Eval in eXide. The following error is raised:

```

module namespace test2 = "test2";
```

Of course, library modules aren't meant to be directly evaluated, but IDEs like eXide do compile library modules to report static errors (and Java admin client happily accepts submission of library modules), so this is a valid use case. In these cases, eXist should not raise this error.

Neither BaseX nor Saxon raise such errors. They both happily validate test2.xqm.

**Context (please always complete the following information):**
 - OS: macOS 10.15.5
 - eXist-db version: 5.2.0, 5.3.0-SNAPSHOT
 - Java Version: JDK 1.8.0_252 (Zulu 8.46.0.19-CA-macosx) (build 1.8.0_252-b14)

**Additional context**
- How is eXist-db installed? 5.2.0 via DMG installer, 5.3.0-SNAPSHOT built from source 
- Any custom changes in e.g. `conf.xml`? None.
