**Describe the bug**

It is not spec compliant to allow 
```xquery
function(xs:long)
function(xs:int)
function(xs:short)
function(xs:byte)
```
to be called with a literal number. 
According to the discussion in this [SO-question]( https://stackoverflow.com/questions/60877203/why-can-a-numeric-literal-never-be-of-type-xspositiveinteger-in-xquery) this is not spec-compliant.

The relevant portion of Michael Kay's answer:

> Then ยง3.1.1 in the XQuery spec talks about numeric literals:
>> The value of a numeric literal containing no "." and no e or E character is an atomic value of type xs:integer.

> ยง3.18.1 gives the rules for the "instance of" operator:
>> The boolean operator instance of returns true if the value of its first operand matches the SequenceType in its second operand, according to the rules for SequenceType matching;

> and ยง2.5.5.2 gives the relevant rule for SequenceType matching:
>> An ItemType consisting simply of an EQName is interpreted as an AtomicOrUnionType. The expected type AtomicOrUnionType matches an atomic value whose actual type is AT if derives-from( AT, AtomicOrUnionType ) is true.


**Expected behavior**

Calling a function with one the above type-signatures with a literal number like `1` should raise an error (`err:XPTY0004`, "cannot cast xs:integer to T"). 

**To Reproduce**

```Xquery
xquery version "3.1";

module namespace t="http://exist-db.org/xquery/test-downcast";

declare namespace test="http://exist-db.org/xquery/xqsuite";

declare function t:long ($n as xs:long) { $n };
declare function t:int ($n as xs:int) { $n };
declare function t:short ($n as xs:short) { $n };
declare function t:byte ($n as xs:byte) { $n };

declare variable t:fun-seq := (t:long#1, t:int#1, t:short#1, t:byte#1)

declare
    %test:param(1)
    %test:param(2)
    %test:param(3)
    %test:param(4)
    %test:assertError('XPTY0004')
function t:test($index) { t:fun-seq[$index](1) };
```
